## 1. 프로젝트 개요
- **프로젝트 명**: Project Zephyr
- **성격**: 게임 및 엔진 임베딩(Embedding)을 타겟으로 설계된, Rust 형태의 문법을 가진 스크립팅 언어.
- **사용 언어**: C++17 이상 추정 (코어 런타임 및 가상머신 구현).
- **빌드 환경**: Visual Studio 기반의 `.sln` (Zephyr.sln) 및 `.vcxproj` 프로젝트 파일로 관리.

## 2. 주요 디렉토리 구성 (Directory Structure)

코드베이스는 다음과 같이 관심사에 맞게 디렉토리가 분리되어 있습니다.

- **`include/zephyr/`**
  - **역할**: 외부 엔진이나 애플리케이션에서 Zephyr 스크립트 런타임을 임베딩할 때 사용하는 공용 C++ API (주로 `api.hpp` 포함).
  - **내용**: `ZephyrVM`, `ZephyrClassBinder`, GC/코루틴 통계 모델 등 인터페이스 정의.
- **`src/`**
  - **역할**: 언어 해석, 파싱, GC 연산 등 핵심 구현부가 위치한 디렉토리.
  - **내용**: `zephyr.cpp` 외에 `zephyr_compiler.inl`, `zephyr_lexer.inl`, `zephyr_parser.inl`, `zephyr_gc.inl` 등 `.inl` 확장자를 띄는 아주 큰 인라인 형식의 모듈 조각들로 구성되어 있어 단일 컴파일(Unity Build) 패턴 또는 빠른 인라인 최적화를 추구한 것으로 파악됩니다.
- **`examples/`**
  - **역할**: `.zph` 확장자를 사용하는 Zephyr 언어의 샘플 스크립트 파일들.
  - **내용**: 상태 머신(`state_machine.zph`), 이벤트 핸들링(`event_handling.zph`), 간단한 AI 제어 구조(`simple_ai.zph`), 기본 문법 테스트용 코드들이 있습니다.
- **`tests/` 및 `zephyr_tests/`**
  - **역할**: 언어 스펙과 GC, 호스트 연동 등에 대한 단위 기능 및 통합 테스트 레이어 모음.
- **`cli/` 및 `bench/`**
  - **역할**: 개발 도구. 커맨드라인 인터페이스 기반 REPL, 구문 검사기, 바이트코드 덤프, 그리고 엔진 성능 척도를 측정하기 위한 JSON 벤치마크 환경.

## 3. 핵심 아키텍처 및 시스템 심층 분석

이 프로젝트는 게임 엔진의 틱(Tick) 주기와 메모리 제약을 극복하기 위해 다수의 자체 시스템을 정교하게 구축했습니다. 주요 시스템을 네 가지(분석 파이프라인, 코루틴, 바인딩, GC)로 나누어 살펴봅니다.

### 3.1. 컴파일러 및 바이트코드 시스템 (Compiler & Bytecode)
- **파서 및 렉서 (Parser & Lexer)**: `zephyr_lexer.inl`과 `zephyr_parser.inl`을 통해 스크립트를 파싱하고 AST(추상 구문 트리)로 생성합니다. Rust 스타일(`struct`, `enum`, `match` 등)의 안전하고 명확한 언어 구조를 엄격하게 파싱합니다.
- **슈퍼인스트럭션 퓨전 (Superinstructions)**: `zephyr_compiler.inl` 내부 구조에서 볼 수 있듯, 잦은 연산 패턴(`SILoadLocalAddStoreLocal`, 조건 분기 등)을 하나의 바이트코드 명령어로 압축(Fusion)하여 인터프리터 루프의 디스패치(Dispatch) 오버헤드를 대폭 줄였습니다.
- **가상 머신 실행 아키텍처**: 단순히 스택 기반 평가만 하는 것이 아니라 `RegisterAllocator`를 통해 가상 레지스터 모드를 도입(`R_MOVE`, `R_ADD` 파생 옵코드 등)하여 런타임에 메모리 읽고 쓰는 비용을 최소화하도록 설계되어 있습니다.

### 3.2. 코루틴 및 비동기 상태 시스템 (Coroutine State Machine)
- **프레임 단위 힙 영속화**: 일반적인 콜스택이 아니라 함수 실행 프레임(`CoroutineFrameState`)과 캡처된 로컬 변수 환경(`UpvalueCell`) 자체를 단일 `CoroutineObject` 내부의 힙 메모리로 보관합니다.
- **안전한 제어 역전 (Yield/Resume)**: 중간에 어떠한 C++ AST 평가 폴백 없이 자체 바이트코드 수준에서 즉각적으로 연산을 일시정지(`yield`)하고, 나중에 엔진 틱(Tick)에서 이어서(`resume`) 실행할 수 있습니다. 수많은 몬스터나 AI 액터를 다루는 게임 로직의 코루틴 스케줄링 처리에 완벽하게 최적화되어 있습니다.

### 3.3. 동적 런타임 및 엔진 바인딩 시스템 (Runtime & ZephyrVM)
- **엄격한 C++ 바인딩 (`ZephyrClassBinder<T>`)**: 제네릭 기반 메타프로그래밍으로 C++ 코드 단의 복잡한 펑터 변환 없이 즉시 클래스의 메서드/프로퍼티를 노출(Export)합니다. 리플렉션이 없는 C++의 한계를 템플릿 바인더로 깔끔하게 해결했습니다.
- **4단계 핸들 정책 (Host Handles Lifecycle)**:
  - 네이티브 포인터의 원시 메모리를 그대로 넘기지 않고 핸들(Handle)로 넘깁니다.
  - **Frame / Tick**: 매 프레임이나 틱 종료 시점에 모두 만료시켜, 일시적인 계산에 쓰이는 객체의 메모리 누수와 매달린 포인터(Dangling pointer)를 방지합니다 (프레임 내/틱 내 안전 객체).
  - **Persistent / Stable**: 씬(Scene)이 유지되는 동안, 혹은 게임 전체 세션/직렬화(Serialization)가 필요한 대상에만 살아있도록 차등 허용 정책을 폅니다.

### 3.4. 고성능 세대별 가비지 컬렉터 (High-Performance Generational GC)
엔진 환경이 요구하는 엄격한 지연 시간(프레임 스파이크 방지) 조건을 위해 매우 정교한 GC(`zephyr_gc.inl`)를 사용합니다.
- **4-스페이스 관리 (4-Space Heap)**: `Nursery`, `Small Old`, `Large Object(LOS)`, `Pinned`의 네 구역으로 나누어 효율적인 할당 및 해제를 관리합니다. 
- **조각 모음 (Compaction)**: `compact_old_generation` 구문을 통해 파편화된 다수의 구형(Old) 객체들을 새 슬랩(Slab) 메모리 슬롯에 재배치하여 장시간 플레이되는 게임 클라이언트의 힙 파편화 문제를 실시간으로 해결합니다.
- **카드 트래킹 (Remembered Cards Tracking)**: 구조체 인스턴스, 배열, 로컬 환경 등 오래된 객체에 새로운 (Nursery) 데이터가 저장될 경우 쓰기 장벽(Write Barrier)이 작동하여 비트맵(Bitmap) 카드에 오염 여부를 기록합니다. 이후 마이너 GC 실행 시 힙 전체가 아닌 오염된 "Card" 범위 내에서만 포인터를 스캔하여 병목 현상(Traversial overhead)을 극단적으로 피합니다.
- **증분 실행 (Incremental Step)**: `gc_step(budget)`을 통해 매 프레임 한정된 작업 예산 내에서만 마킹과 청소 로직을 분산 실행하기 때문에 플레이어가 끊김(Hitch)을 체감하지 않도록 합니다.

## 4. 버그 가능성, 한계 및 단점 (Limitations & Disadvantages)

현재 프로젝트 아키텍처 상 발생할 수 있는 주요 버그 가능성과 시스템적 한계는 다음과 같습니다.

### 4.1. 구조적 한계와 커스텀 바이트코드 제약
- **단일 컴파일(Unity Build) 지향**: 주요 로직이 거대한 `.inl` 파일들에 집중되어 있어 모듈성이 떨어집니다. 코어 엔진 소스 수정 시 컴파일 시간이 매우 길어질 수 있는 아키텍처적 단점이 있습니다.
- **레지스터 할당 한계 (8-bit Register Limit)**: 바이트코드 인터프리터가 최적화를 위해 (24바이트 구조 강제로 인해) 최대 256개의 지역 변수(임시 레지스터)만 허용하도록 하드코딩 되어 있습니다.
  - *해결 방안(에뮬레이션 우회)*: 256개 한계 초과 시 힙 기반의 로컬 배열(Spill) 스택을 사용하여 명령어를 평가하도록 컴파일러 레벨에 "느린 우회 경로(Fallback path)" 규칙을 추가하는 구조적 개선이 필요합니다.
- **JIT(Just-In-Time) 컴파일 기능 부재**: 전용 바이트코드 인터프리터만 제공되어 수학 연산이 극도로 많은 로직의 경우 LuaJIT나 V8 엔진에 비해 절대적인 싱글스레드 퍼포먼스 제약이 분명합니다.

### 4.2. 메모리 핸들 및 GC의 취약성
- **엄격한 C++ 바인딩 생명주기 의존성**: 잘못된 수명 정책(`Tick` 또는 `Frame` 핸들 등)을 부여하거나 가비지 컬렉터에게 루트 참조를 제대로 동기화하지 않으면 댕글링 포인터(Use-After-Free)가 발생하기 쉽습니다.
- **객체 파편화 모음(Compaction) 버그 위험성**: 조각 모음(`compact_old_generation`) 시 메모리 주소가 이동할 때, C++ 측에서 이동 전 원시 포인터를 참조하고 있다면 심각한 동기화 오류 크래시가 발생할 수 있습니다.
  - *해결 방안 후보 정리*:
    1. **간접 핸들 테이블 (Handle Table Indirection)**: C++에 원시 포인터가 아닌 테이블 ID(인덱스)만 발급하여, 객체 이동 시 테이블 내부 주소만 갱신하는 인프라스트럭처 도입.
    2. **임시 고정(Pinning) API 추가**: `ZephyrVM::pin(obj)` API 등을 통해, 해당 객체가 GC 이동 대상에서 완전히 제외되는 Pinned Space에 고정되도록 강제하는 방식.
    3. **RAII 기반 스마트 핸들**: `ZephyrHandle<T>`와 같은 C++ 스마트 포인터 래퍼 내부에서 자동으로 Root Set에 객체를 얽고 수명 주기에 따라 해제하도록 구현하여 휴먼 에러 방지.

### 4.3. 코루틴 경계 한계
- **C++ 경계 크로스 Yield 불가**: Zephyr 코루틴은 자체 힙 스택 프레임 내에서만 상태를 유지합니다. 따라서 Zephyr -> C++ -> Zephyr 로 중첩 호출된 깊은 스택 구간에서는 `yield`를 안전하게 사용하여 최상단 시스템을 중지시킬 수 없는 한계가 있습니다. C++ 자체의 네이티브 콜스택은 보존할 수 없기 때문입니다.

### 4.4. 생태계 및 도구 (Ecosystem)
- **디버깅 도구의 부족**: 내부적으로 `dap_server`(Debug Adapter Protocol)를 지원하는 코드는 있으나, 범용적인 언어가 아니므로 IDE(VSCode 등) 완벽 지원이나 중단점(Breakpoint), 스택 백트레이스 도구가 아직 매우 제한적일 수밖에 없습니다. 버그 발생 시 스크립트와 C++의 경계를 넘나들며 디버깅하기가 어렵습니다.

## 5. 결론

Project Zephyr는 범용 스크립팅보다 **게임 엔진 연동(Game Engine Embedding)에 최우선 목표를 둔, 가볍고 빠르며 안전한 언어**입니다.
C++ 수준에서는 최신 C++17 이상 기능을 활용하여 바인딩이 매우 쉽도록 템플릿 형태로 제공되며, 언어 사용자 측면에서는 제어 역전이 쉬운 코루틴 및 Rust와 동일한 가독성 높은 구문 덕분에 C#이나 Lua 등을 대체할 수 있는 매력적인 스크립팅 레이어로 평가됩니다. 
다만, 완전한 커스텀 언어인 만큼 8-bit 레지스터 컴파일 한계, 엄격한 메모리 핸들의 러닝커브, 부족한 IDE 디버깅 생태계 등 실무 도입 및 확장 시 극복해야 할 시스템적, 도구적 한계 또한 명확히 남아있습니다.
